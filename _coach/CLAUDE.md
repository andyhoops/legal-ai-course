# Coach Context — Legal AI Learning Project

*This file is for use in the coach's own Claude Code sessions (planning, review, etc.). The student-facing CLAUDE.md is at the repo root.*

## Project Purpose

This project supports a structured 4-month learning programme for a 16–17 year old student. The goal is for the student to develop practical AI skills — specifically using Claude Code — in the context of UK contract law and legal case analysis.

By the end of Month 4 the student will have:
- A GitHub repository demonstrating their full learning journey
- A GitHub Pages site they can link from their CV
- A personal prompt library for legal research
- Experience using AI as a peer learning partner rather than a shortcut

## Student Profile

- Age: 16–17
- Tech comfort: High (confident with computers, smartphones, digital tools)
- Coding experience: None — treat terminal commands and programming concepts as new
- Legal background: None assumed — all legal content should be introduced from scratch

## Legal Domain

**UK Contract Law** is the anchor domain. Key topics and landmark cases:

**Formation of contract:**
- Offer & acceptance: *Carlill v Carbolic Smoke Ball Co* [1893]
- Invitation to treat: *Fisher v Bell* [1961], *Pharmaceutical Society v Boots* [1953]
- Consideration: *Chappell & Co v Nestlé* [1960], *Williams v Roffey Bros* [1991]
- Intention to create legal relations: *Balfour v Balfour* [1919], *Merritt v Merritt* [1970]

**Terms and performance:**
- Exclusion clauses: *L'Estrange v Graucob* [1934], *Curtis v Chemical Cleaning* [1951]
- Implied terms: Sale of Goods Act / Consumer Rights Act 2015

**Breach and remedies:**
- Remoteness of damage: *Hadley v Baxendale* [1854], *The Achilleas* [2008]
- Mitigation, specific performance, injunctions

**Vitiating factors:**
- Misrepresentation, mistake, duress, undue influence

Always use neutral, accurate legal language. Cite cases with correct neutral citations. When uncertain, say so clearly.

## Session Conventions

When working with the student in Claude Code sessions:
- **Start each session** by reading the student's weekly journal entry if one exists
- **Suggest journal prompts** at the end of sessions — the journal is the backbone of the GitHub portfolio
- **Celebrate small wins** — this is a student, not a professional
- **Explain jargon** — both AI jargon and legal jargon should be unpacked
- **Encourage scepticism** — always invite the student to question or push back on Claude's output
- **Link to the portfolio** — regularly remind the student that good work here will go on their GitHub

## Facilitator Notes

- **Facilitator:** The coach (adult supervisor / course designer)
- **Cadence:** ~1 session per week, 60–90 minutes
- **Platform:** Claude Code (claude.ai/code or CLI)
- **Version control:** Introduce GitHub gently in Month 1 (web interface only), then use Claude Code's git integration by Month 3
- **Assessment:** Reflective rather than exam-based — the journal and portfolio are the evidence of learning

See `session-guide.md` for detailed session structure and tips.

## AI Capability Landscape — Note for the Coach

AI capabilities are evolving rapidly and are not uniform across tools or versions. When running sessions, be aware:

- **Claude Code may or may not have web search** depending on the version and plan. Check before each cohort starts, and flag it to the student.
- **Even with web access, AI can hallucinate.** The Week 8 critical evaluation session addresses this directly — the lesson is about verification habits, not a fixed list of AI limitations.
- **If capabilities have changed since this course was written**, treat that as a teaching moment, not a problem. The student discovering that Claude can now do something it "couldn't before" is excellent material for a journal entry about the pace of AI development.
- **BAILII (bailii.org)** remains the recommended free, authoritative source for verifying UK case citations regardless of Claude's web access status.
